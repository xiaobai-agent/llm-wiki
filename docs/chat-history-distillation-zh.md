# 聊天记录蒸馏：从对话到知识

> **一句话总结**：不存原始聊天记录，而是将其蒸馏为结构化知识——可检索、可被未来 Agent 继承、压缩率 90%+ 同时保留核心价值。

## 问题背景

AI Agent 会随时间积累大量对话历史：
- 上下文压缩导致早期讨论被遗忘
- 有价值的洞察散落在日常闲聊中
- 跨会话检索困难
- 知识无法迁移给新 Agent

## 业界共识

主流记忆框架（Mem0、Zep、Letta、Cognee）的一致做法：

> **不存原始对话，存提炼后的知识。**

```
原始对话 → 提取原子化事实 → 嵌入向量 → 存储/检索
```

### AWS AgentCore 研究数据

| 记忆类型 | 准确率 | 压缩率 |
|---------|-------|--------|
| 语义记忆 | 70-74% | 89-94% |
| 偏好记忆 | 79% | 68% |
| **摘要记忆** | **83%** | **95%** |

**结论**：摘要记忆可实现 95% 压缩率 + 83% 准确率，是最佳平衡点。

## 架构：三层蒸馏模型

```
┌─────────────────────────────────────────────────┐
│  原始聊天记录（备份，保留在原平台）                   │
└─────────────────────────────────────────────────┘
                        ↓ 导出
┌─────────────────────────────────────────────────┐
│  JSON/JSONL 格式（本地临时存储）                     │
└─────────────────────────────────────────────────┘
                        ↓ 按周处理
┌─────────────────────────────────────────────────┐
│  Layer 1: 时间线摘要                                │
│  - 粒度：按周                                       │
│  - 内容："这周主要聊了什么"                          │
│  - 存储：wiki/timelines/2024-W01.md                │
└─────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────┐
│  Layer 2: 原子化事实                                │
│  - 粒度：单条事实                                   │
│  - 内容：用户的观点、决定、偏好                      │
│  - 存储：追加到相关概念页                            │
└─────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────┐
│  Layer 3: 未落地想法                                │
│  - 粒度：想法/可能性                                │
│  - 内容：讨论过但没执行的                            │
│  - 存储：wiki/ideas/backlog.md                     │
└─────────────────────────────────────────────────┘
```

## 两种模式：历史批量 + 增量实时

### 模式一：历史批量处理（一次性）

针对已有的聊天历史：

1. **导出**：通过平台 API 导出为 JSONL
2. **切片**：按周分割
3. **蒸馏**：LLM 提取摘要、事实、想法
4. **审核**：人工抽查
5. **入库**：写入 Wiki 结构

**预估成本**：2GB 聊天记录约 $30-50

### 模式二：实时蒸馏（持续）

针对每日新增对话——**关键创新**：

#### 方案 A：批量（每周）
```
每周日 → 导出本周 → LLM 蒸馏 → 入库
```
- 延迟：1 周
- 简单但不够新鲜

#### 方案 B：每日批量
```
每天凌晨 → 导出昨日 → LLM 蒸馏 → 入库
```
- 延迟：1 天
- API 成本略高

#### 方案 C：实时（推荐 ⭐）

Agent 在对话中**主动识别**值得记录的内容，当场写入。

**触发条件**：
- 用户做了明确决定
- 讨论出新结论/洞察
- 用户表达重要偏好
- 提到新的项目/人物/概念

**优势**：
- 零延迟——对话中即入库
- 上下文完整——当场理解意图最准确
- 无额外成本——不需要单独跑脚本
- 用户可控——随时说"把这个记一下"

#### 推荐：C + A 组合

- **日常**：实时蒸馏
- **兜底**：每周日复盘查漏 + 生成时间线摘要

## Prompt 模板

### 时间线摘要

```
你是一个知识整理助手。以下是某一周的聊天记录。

请输出：
1. **本周主题**：列出 3-5 个主要话题
2. **关键进展**：有什么决定/结论（用户和 AI 双方）
3. **待跟进**：提到但没完成的事项

注意：用户和 AI 的发言都要关注：
- 用户的决定、需求、偏好
- AI 的调研结论、技术方案、经过验证的发现

格式：Markdown，简洁，每个话题 1-2 句话。
保留具体的项目名/人名/日期。

聊天记录：
{messages_json}
```

### 原子化事实提取

```
你是一个事实提取器。从以下对话中提取原子化事实，**用户和 AI 双方都要关注**：

**从【用户】提取**：
- 明确的决定和观点
- 表达的偏好
- 重要信息（人物、项目、数字）

**从【AI】提取**：
- 经过验证的技术发现（如：某 API 不支持 XX）
- 已确认有效的解决方案（如：用 PowerShell 而不是 curl 处理 multipart）
- 已确认的架构设计决定
- 踩坑经验和规避方法

要求：
1. 每条事实独立成句，脱离上下文也能理解
2. AI 的事实只提取"经过验证的结论"，不提取猜测或初步想法
3. 保留具体细节（数字、名字、日期、API 名称）
4. 用 source 字段标明来源："user" 或 "ai"
5. 如果没有值得提取的，返回空列表

输出 JSON：
{
  "facts": [
    {"fact": "...", "date": "2024-01-15", "topic": "...", "source": "user"},
    {"fact": "...", "date": "2024-01-15", "topic": "...", "source": "ai"},
    ...
  ]
}

聊天记录：
{messages_json}
```

### 未落地想法

```
你是一个想法收集器。从以下对话中找出未落地想法，**用户和 AI 双方都要关注**：

**从【用户】收集**：
- 讨论过但没有执行的想法
- 备选方案
- 标记为"以后再说"的事项

**从【AI】收集**：
- AI 提出但用户没采纳的建议
- 调研发现的潜在方向但没深入的
- AI 给出的备选方案（用户选了 A，但 B 方案也值得记录）
- 技术可行但暂时没精力做的优化建议

要求：
1. 只收集未执行的，已执行的不算
2. AI 的建议要区分"被采纳"和"被搁置"，只记后者
3. 用 source 字段标明来源："user" 或 "ai"

输出 JSON：
{
  "ideas": [
    {"idea": "...", "context": "...", "date": "...", "source": "user"},
    {"idea": "...", "context": "...", "date": "...", "source": "ai"},
    ...
  ]
}

聊天记录：
{messages_json}
```

## 目录结构扩展

```
wiki/
├── pages/           # 现有

├── meta/            # 现有
├── timelines/       # 新增 - 周报摘要
│   ├── 2024-W01.md
│   ├── 2024-W02.md

├── ideas/           # 新增 - 未落地想法
│   └── backlog.md
└── index.md
```

## 质量保证

### 人工审核（轻度）

| 层级 | 审核方式 | 耗时 |
|------|---------|------|
| 实时入库 | Agent 说"记了"，用户瞄一眼 | ~0 |
| 周报 | Agent 发摘要，用户快速浏览 | 5分钟/周 |
| 历史批量 | Agent 发汇总，用户抽查 | 30分钟（一次性）|

**原则**：信任 Agent 判断，有疑问时 Agent 主动询问。

### 去重

基于语义相似度去重，避免重复入库。

### 版本控制

所有蒸馏结果通过 Git 管理，可回滚任意版本。

## 安全考量

### 敏感信息过滤

蒸馏前自动过滤：
- API Key / Token
- 密码
- 身份证号、银行卡号
- 私人联系方式

### 数据保留策略

| 数据类型 | 位置 | 保留策略 |
|---------|------|---------|
| 原始聊天记录 | 原平台 | 永久（平台保留）|
| 导出 JSONL | 本地临时 | 处理后删除 |
| 蒸馏结果 | Wiki + Git | 永久 + 版本控制 |
