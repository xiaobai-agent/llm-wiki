## 1. 引言与灵感来源

### 什么是 LLM Wiki？

LLM Wiki 是一个由 LLM Agent 维护的个人知识库系统。与传统的检索增强生成（RAG）不同，LLM **将知识一次性编纂并持续积累** —— 每一条新信息都在丰富已有的知识图谱，而不是每次查询时重新发现。

### 灵感来源

本项目直接受到 [Andrej Karpathy 的 llm-wiki 概念](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)（发布于 2026-04-04）的启发：

> RAG rediscovers knowledge every time with no accumulation. A wiki is a continuously compounding knowledge artifact.
>
> （RAG 每次都要重新发现知识，没有积累。而 Wiki 是一个持续复利增长的知识产物。）

Karpathy 提出了三层架构（Raw Sources → Wiki Pages → Schema）和三种核心操作（Ingest、Query、Lint）。我们完整采纳了这一框架，并根据自身环境适配了工具链。

这个理念可以追溯到更早——**Vannevar Bush 在 1945 年发表的 "As We May Think"** 以及其中提出的 **Memex** 概念：一台私人知识设备，文档之间的关联与文档本身同样重要。81 年后，LLM Wiki 实现了这一愿景，而 LLM 解决了那个关键问题——"由谁来维护？"

### 为什么要做这个？

开发者的核心痛点很简单：**有价值的内容收藏了就丢了。** 知识散落在社交媒体收藏、浏览器书签、聊天记录、笔记应用等各处——没有一个统一的系统来存储、关联和检索。

目标：**一个"放进去就能找得到"的系统**，并由 AI 来理解上下文、建立关联、主动维护知识库。

## 2. 需求分析

### 2.1 核心痛点

- 社交媒体上收藏的有价值内容，之后几乎找不到
- 知识分散在多个平台，缺乏统一管理
- 需要一个真正能"存得进、找得到"的系统

### 2.2 需求确认

| 问题 | 回答 |
|------|------|
| 内容来源 | 社交媒体文章、新闻、短视频平台、PDF、个人笔记 |
| 主要设备 | 手机 + 平板（电脑很少用） |
| 检索方式 | 对话式 —— 自然语言提问，不需要想关键词 |
| 维护模式 | 人机协作：用户收集素材，AI 整理和维护 |

### 2.3 长期愿景

**两阶段演进：**

1. **第一阶段：** 用户是主角，AI Agent 是超级助手 —— 用户收集素材、提出问题，AI 整理维护 Wiki
2. **第二阶段：** AI Agent 是主角，用户是超级助手 —— AI 自我进化，主动发现和探索知识

**终极目标：** Wiki 不只是一个书签收藏夹——它是 AI Agent 的知识底座和进化引擎。

### 2.4 输入设计

- **第一步（人工投喂 + AI 整理）：** 用户将内容发给 AI → AI 分析、提取、整理进知识库
- **第二步（AI 主动发现）：** AI 分析用户的信息消费习惯 → 主动监控信息源 → 发现有价值的内容 → 提醒用户

### 2.5 检索层设计

前端 = 即时通讯界面，用户用自然语言描述要找的东西 → AI 在知识库中进行语义检索 → 返回结果 → 必要时补充外部搜索。

## 3. 架构设计

### 3.1 设计原则

在研究了 GitHub 上多个 Karpathy llm-wiki 实现方案后，我们确立了以下原则：

1. **核心理念源自 Karpathy** —— 三层架构 + 三种操作
2. **工具链适配自身环境** —— 用聊天界面替代 Obsidian + 命令行
3. **不重复造轮子** —— 充分利用 AI Agent 平台已有的基础设施
4. **移动端优先** —— 所有操作都能通过手机聊天完成

### 3.2 架构概览

![架构示意图](architecture-diagram.png)

*注：架构图展示了三层结构及数据在 Raw Sources、Wiki Pages 和 Schema 之间的流转，并通过聊天界面与用户交互。*

#### 第一层：Raw Sources（原始素材）

- **位置：** `wiki/raw/` —— 原始文章、PDF 文本、笔记、视频字幕
- **规则：** 只读。AI 永远不修改原始素材。这是事实来源（Source of Truth）。

#### 第二层：Wiki Pages（知识页面）

- **位置：** `wiki/pages/` —— 概念页、实体页、素材摘要页、对比分析页、洞察页
- **索引：** `wiki/index.md`（目录）+ `wiki/log.md`（操作日志）

#### 第三层：Schema（规则文件）

- **位置：** `WIKI-SCHEMA.md` —— 结构规范、页面模板、工作流定义

#### 交互层：聊天界面

- **输入：** 用户通过聊天向 AI Agent 发送链接/文字/截图/视频
- **输出：** AI 在对话中回复，或生成文档

### 3.4 页面格式模板

每个 Wiki 页面必须以 YAML Frontmatter 开头：

```yaml
---
title: "页面标题"
type: concept | entity | source | comparison | insight
tags: [tag1, tag2]
sources: [wiki/raw/path/to/source.md]
related: [wiki/pages/path/to/related.md]
created: 2026-04-06
updated: 2026-04-06
confidence: high | medium | low
---
```

正文使用标准 Markdown 格式，通过内部链接引用其他 Wiki 页面。

### 3.5 三种核心操作

#### 操作一：Ingest（入库，7 步）

**触发条件：** 用户在发送链接/文字/截图/视频时，包含触发关键词。

**工作流：**

1. **获取原始内容**（网页抓取 / 直接文本 / 图片识别 / ffmpeg + 语音转文字 API）
2. **保存到 `raw/`** 对应的子目录
3. **与用户讨论要点**（可选）
4. **创建/更新 Wiki 页面**（sources/ + concepts/ + entities/）
5. **更新 `index.md` 和 `log.md`**
6. **向用户确认完成**
7. **更新 `meta/`**（如果有新的策略或经验教训）

> 一份素材可能涉及 5-15 个 Wiki 页面。

#### 操作二：Query（检索，6 步）

**触发条件：** 用户用自然语言提出任何检索问题。

**工作流：**

1. 读取 `index.md` 定位相关页面
2. 辅以语义检索（memory_search）补充
3. 阅读匹配的页面
4. 综合回答并标注信息来源
5. 如果回答有价值，询问是否需要保存为洞察页
6. 当引用 3 个及以上来源时，主动建议将分析写入 `insights/`

#### 操作三：Lint（质检，6 项检查）

**触发条件：** 用户说"检查知识库"或定时执行。

**检查项：**

1. 页面之间的矛盾信息
2. 孤岛页面（没有任何入链）
3. 被提及但尚未创建的概念页
4. 已被更新资料取代的过时信息
5. 应当补充的知识空白
6. Frontmatter 完整性校验

### 3.6 技术栈对照

| Karpathy 方案 | 我们的方案 | 选择原因 |
|---|---|---|
| Obsidian（浏览 Wiki） | 聊天界面 + 在线文档 | 适合移动端 |
| Claude Code（命令行） | AI Agent（对话式） | 不需要命令行 |
| 本地 Markdown 文件 | 工作区 Markdown 文件 | 自动备份到 GitHub |
| qmd 搜索引擎 | 语义记忆检索 | 已有现成能力 |
| Git 版本控制 | 自动化备份脚本 | 每日运行，推送到 GitHub |
| Obsidian Web Clipper | 用户通过聊天发送 + AI 抓取 | 适合移动端工作流 |

## 4. 实施步骤与迭代过程

### 4.1 初始搭建（2026-04-06 13:20）

1. ✅ **创建目录结构** —— `wiki/raw/` 含 7 个子目录，`wiki/pages/` 含 5 个子目录
2. ✅ **编写 WIKI-SCHEMA.md v1.0** —— 完整规则手册，约 7KB
3. ✅ **初始化索引** —— 空的 `index.md`，`log.md` 写入初始化记录，`sources.md` 用于来源追踪

### 4.2 首次 Ingest 测试（2026-04-06 13:29）

- **素材：** 一篇关于 Karpathy LLM Wiki 的新闻报道
- **产出：** 1 份原始素材 → 3 个 Wiki 页面（素材摘要 + 概念页 + 实体页）
- **结果：** 索引和日志均已更新。流程验证通过。**知识库正式上线。** 🎉

### 4.3 触发词约定

一个关键的早期决策——AI 如何区分"需要入库"和"只是闲聊"？

- **入库操作：** 消息中必须包含触发词 "wiki" —— 写操作需要明确授权
- **检索操作：** 无需前缀 —— 用户自然提问，AI 自动判断从哪里找答案
- **质检操作：** 说"检查知识库"即可触发
- **优先级链：** Wiki → 记忆 → 外部搜索

> 没有触发词时，AI **绝不** 自主向 Wiki 添加内容。这防止了知识库被闲聊信息污染。

### 4.4 素材类型约定（2026-04-07 新增）

一个惨痛教训：当用户说 "wiki 视频 [标题]" 时，AI 跑去全网搜这个视频，而不是等用户发文件——浪费了大量 Token。

**解决方案 —— 素材分两类：**

| 类型 | 触发方式 | AI 的行为 |
|------|---------|-----------|
| 视频文件 | "wiki 视频" + 标题 | **等文件！** 记录标题，不要上网搜 |
| 文档文件 | "wiki 文档" + 描述 | **等文件！** 记录描述，不要提前处理 |
| URL 链接 | "wiki" + URL | 立即处理 —— 抓取并入库 |
| 文本/笔记 | "wiki" + 文本 | 立即处理 —— 直接存储 |

**铁律：** 看到 "wiki 视频" 或 "wiki 文档"，AI 必须等用户发文件。绝对不要上网搜！

## 5. 多模态内容接入

### 5.1 视频内容 —— 挑战所在

大量有价值的知识来自抖音、YouTube Shorts 等短视频平台。视频内容无法直接以文本形式存入 Wiki，需要一套完整的流水线实现：视频 → 文本 → 知识。

### 5.2 技术验证

**语音转文字 API 测试（2026-04-06）：**

- 模型：`gpt-4o-mini-transcribe`（云端 API）
- 测试：10 秒中文音频
- 结果：完美转写，标点符号正确
- Token 消耗：229 音频 + 69 文本 = 共 298
- **成本估算：30 分钟视频约 $0.01 美元** —— 几乎可以忽略

**服务器 GPU 检查：** 没有独立 GPU → 云端 API 是最优路径（无需本地部署 Whisper）。

### 5.3 视频入库流水线

**用户的操作流程（手机上 3 步）：**

1. 从平台下载视频
2. 通过聊天将视频文件发给 AI，附带 "wiki" 关键词
3. 等待 AI 处理完成

**AI 的处理流水线：**

```
接收视频 → ffmpeg 提取音频（mp3, 16kHz, 单声道, 64kbps）
→ 调用语音转文字 API → 获取文字稿
→ 保存到 wiki/raw/video/YYYY-MM-DD-[平台]标题.md
→ LLM 分析文字稿，提取知识
→ 创建/更新 Wiki 页面
→ 更新 index.md 和 log.md
→ 向用户确认完成
```

### 5.4 实测数据

| 视频 | 时长 | 文件大小 | 音频大小 | 文字稿 | 成本 |
|------|------|---------|---------|--------|------|
| 关于 Token 经济学的技术分享 | 4:31 | 27.5 MB | 2.07 MB | 1,383 字 | ~$0.002 |
| 算法开源讨论 | 3:43 | 22.6 MB | 1.70 MB | 1,236 字 | ~$0.002 |
| 源码解析分享 | 3:57 | 29.6 MB | 1.81 MB | 1,852 字 | ~$0.002 |

**关键发现：** ffmpeg 参数 `16kHz 单声道 64kbps` 是语音转写的最佳性价比配置——文件极小且不损失精度。30MB 的视频压缩成约 2MB 的音频（16:1 压缩比），处理耗时不到 2 秒。

### 5.5 两条消息的交互问题

**发现的问题：** 聊天平台不允许在同一条消息中同时发送文字和视频。视频只能作为独立消息发送。这意味着用户无法在一条消息中同时包含 "wiki" 触发词和视频文件。

**解决方案：两条消息关联**

- 默认（推荐）：用户直接发送视频文件。AI 转写后根据内容自动生成标题和分类。用户可以在前后补充上下文信息。
- 增强模式：用户先发 "wiki [平台] 标题"，再发视频。AI 在 2 分钟窗口内自动关联消息。

### 5.6 视频归档策略

**问题：** 服务器存储有限（约 20GB）。已转写的视频不应长期占用本地磁盘。但用户偶尔需要"把那个视频发给我"。

**解决方案：热存储 + 冷存储双轨制**

- **热存储（主用）：** 云盘 —— 用于日常检索和分享。用户可以即时获取链接。
- **冷存储（备份）：** GitHub Releases —— 免费、容量无限、永久保留。仅在热存储失效时使用。

**为什么选择 GitHub Releases？**
- 单文件限制：2 GiB（完全够用）
- 总存储量：无限制，免费
- 带宽：无限制
- 保留期限：永久
- 私有仓库：支持
- 对比 Git LFS：LFS 只有 1GB 免费，超出后 $5/50GB/月

**媒体索引文件：** `wiki/media-index.md` 充当所有大文件的查找表，包含日期、标题、类型、大小、时长、归档位置、文字稿路径和 Wiki 页面路径等字段。

**核心理念：** *文本留本地（快速检索），大文件存远端（节省空间），索引连接两端。*

### 5.7 网页文章入库

网页文章的流水线更简单：

```
用户发送 URL + "wiki" → AI 抓取全文（trafilatura）
→ 保存到 wiki/raw/[来源]/ → 提取知识
→ 创建/更新 Wiki 页面 → 更新索引 → 确认完成
```

**关键设计决策：**
- 去重检测：检查已有文件 Frontmatter 中的 URL 字段，相同 URL 跳过
- 短内容检查：正文不足 100 字符 = 自动判定失败，建议使用浏览器自动化处理 JS 渲染页面
- 依赖自安装：脚本自行安装所需依赖包（解决上下文压缩后"忘了已安装"的问题）

## 6. Schema 演进（v1.0 → v1.3）

Schema（`WIKI-SCHEMA.md`）在 48 小时内经历了四次迭代。每个版本都是在实际使用中发现真实问题后演进的。

### v1.0（2026-04-06）—— 首版发布

**内容：**
- 三层架构定义（Raw / Pages / Schema）
- 三种核心操作（Ingest / Query / Lint）
- 触发词约定（"wiki" = 入库，自然语言 = 检索）
- 页面格式规范（YAML Frontmatter）
- 文件命名约定
- 7 个 raw 子目录、5 个 pages 子目录
- 核心规则：raw/ 只读、pages/ 由 AI 维护、始终记日志、始终更新索引

### v1.1（2026-04-06）—— 视频支持

**变更内容：**
- 新增 `wiki/raw/video/` 目录，统一存放全平台视频字幕（不再按平台分目录）
- 文件命名带平台前缀：`YYYY-MM-DD-[平台]标题.md`

**变更原因：**
- 最初计划为每个视频平台设置独立目录。实现过程中意识到：不管来自哪个平台，处理流水线完全相同（提取音频 → 转写 → 入库）。用文件名前缀和 Frontmatter 标签来追踪来源，比目录结构更灵活。
- 新增一个平台 = 零成本（无需新建目录）

### v1.2（2026-04-06）—— 视频归档与媒体索引

**变更内容：**
- 新增视频源文件归档策略（GitHub Release 冷备份 + 云盘热检索）
- 新增 `wiki/media-index.md` 用于大文件追踪
- 定义完整的 存储 → 检索 → 发送 工作流

**变更原因：**
- 服务器存储空间吃紧（剩余约 20GB）
- 需要一种方式归档视频源文件，同时保持可检索
- GitHub Releases 为此场景提供了无限免费存储

### v1.3（2026-04-08）—— 自我进化模块

**变更内容：**
- 新增 `wiki/meta/` 目录，含 5 个策略文件（操作记忆）
- 激活 `wiki/pages/insights/` 用于查询回写
- 新增 Rule 5（策略记忆）和 Rule 6（查询回写）
- Ingest 工作流扩展至 7 步（新增：更新 meta/）
- Query 工作流扩展至 6 步（新增：引用 3 个以上来源时建议回写洞察页）
- 新增 3 个 raw 子目录：`research/`、`doc/`、`web/`
- 新增素材类型约定（两步式 vs. 一步式素材）

**变更原因：**
- 对学术论文（OpenSpace、EvoScientist）的调研揭示了一个问题：我们的系统有知识记忆，但缺少**策略记忆** —— 它记住了"知道什么"，却不记得"怎样做最有效"
- 查询回写的灵感来自 Karpathy 的观点：好的分析性回答应当以洞察页的形式写入 Wiki，而不是消失在聊天记录中

### 7.4 三个增强模块

**核心原则：** 全部用 Markdown 文件实现。不写代码、不改架构。

#### 模块一：查询回写（激活 `insights/` 目录）

- **问题：** 综合多来源的深度分析回答消失在聊天记录中
- **方案：** 将提炼后的分析存入 `wiki/pages/insights/`
- **触发条件：** 当回答引用 3 个以上来源或产生新洞察时，建议保存
- **格式：** `type=insight`，Frontmatter 中的 sources 指向引用的页面
- **命名：** 按主题命名（如 `cooperation-feasibility-analysis.md`），不加日期前缀
- **成本：** 几乎为零——多一个 .md 文件 + 更新索引

#### 模块二：策略记忆（新建 `meta/` 目录）

- **问题：** Wiki 记住了"知道什么"，却不记得"怎样做最有效"
- **方案：** 创建 `wiki/meta/`，包含 5 个策略文件：

| 文件 | 用途 |
|------|------|
| `ingest-strategies.md` | 各类来源的最佳抓取策略 |
| `research-patterns.md` | 调研路径模板 |
| `source-quality.md` | 来源质量评级 |
| `failure-log.md` | 踩坑和失败记录 |
| `skill-playbook.md` | 技能使用技巧与最佳实践 |

- **写入时机：** 每次 Ingest 后，如有新经验教训则更新 meta/
- **读取时机：** 每次 Ingest 前，检查 meta/ 中已有的策略

#### 模块三：技能知识融合

- 在 `meta/skill-playbook.md` 中记录技能使用经验
- 每个技能一个章节：注意事项、最佳实践、实测数据
- 建立从知识到技能的单向通道

## 8. 容量规划与升级路线图

### 8.1 当前架构上限

**当前技术栈：** Markdown 文件 + 文件系统目录 + `index.md` 线性索引

| 维度 | 现状 | 瓶颈 |
|------|------|------|
| 存储 | 无压力 | 10,000 篇文章 ≈ 100MB |
| 检索 | 约 200-300 页 | `index.md` 过大，消耗 Token |
| 理解 | 上下文窗口有限 | 约 150K Token |
| 关系 | 手动维护 Frontmatter `related` 字段 | 无法做结构化关系推理 |

### 8.2 第一阶段：Vector Search（约 200 页触发）

**做什么：** 在 `wiki/` 目录上增加语义向量检索 + 将 `index.md` 拆分为子索引。

- 舒适容量：~200 → ~2,000-3,000 页
- 检索从线性扫描变为语义检索——大幅节省 Token
- 实现成本：极低（可能只需一个配置变更）
- **预计触发时间：** 约 2-3 个月后

### 8.3 第二阶段：本地向量数据库（约 2,000 页，可选）

**做什么：** 部署轻量级向量数据库（ChromaDB 或 Qdrant），每次入库同步写入向量存储。

- 能力提升：标签过滤、时间排序、相似度阈值、批量分析
- 实现成本：中等
- **预计触发时间：** 6-12 个月。可能直接跳到第三阶段。

### 8.4 第三阶段：知识图谱（约 2,000-3,000 页）

**为什么需要知识图谱？**

向量检索能找到"文本相似的页面"，但无法做到：
- 理解实体之间的结构化关系（Company A → 位于 → 城市 X → 属于 → 省份 Y）
- 多跳推理（Company A 有省份 Y 的资质 → 省份 Y 在试点名单中 → Company B 可以跨区域合作）
- 路径发现（Company A 和 Company B 之间有多少条关联路径？）

**知识图谱 = 关系网络。** 节点是实体/概念，边是关系。

**技术选型：**
- 起步用 **NetworkX**（Python 内存图库）—— 无需安装数据库，可处理数千节点，序列化为 JSON 方便 Git 备份
- 按需升级 **Neo4j** —— 专业图数据库，适合 10K+ 节点，Cypher 查询语言
- 考虑 **GraphRAG**（微软开源）—— 从文本中自动提取实体和关系，用图结构增强检索。与我们的场景最契合。

**工作原理：**
- **入库时：** LLM 自动提取实体和关系 → 写入图（NetworkX + JSON）→ Markdown 页面照常生成
- **查询时：** 向量检索找到相关文档 + 图做关系推理 → 综合回答
- **质检时：** 发现孤立节点、矛盾关系，建议潜在的新关联

**预计触发时间：** 1 年以上。

### 8.5 升级总览

```
当前（约 100 页）    → Markdown 文件 + index.md
2-3 个月（约 500 页） → + Vector Search（对 wiki/ 做语义检索）
6-12 个月（约 2000 页）→ + 知识图谱（NetworkX + JSON）
1 年以上（3000+ 页）  → 可能升级到 Neo4j + GraphRAG
```

### 8.6 关键设计决策

1. **Markdown 永不过时** —— 人类可读、Git 可备份、灾难可恢复。它是所有上层索引构建的基础。
2. **先简单后复杂** —— 远未触及瓶颈时，不引入复杂架构。
3. **升级触发标准 = "感受到痛苦"** —— 不是绝对的页面数量，而是实际操作变慢、Token 浪费明显。
4. **每次升级是叠加，不是替换** —— Markdown 文件始终是最终真相来源。
5. **对用户透明** —— 用户的操作流程永远不变（通过聊天发消息），只是后端检索能力在提升。

## 9. 踩坑总结

### 9.1 不要去搜即将送上门的东西

**头号 Token 浪费事故：** 当用户说 "wiki 视频 [标题]" 时，AI Agent 跑去全网搜这个视频（视频网站、论坛、GitHub……）—— 烧掉大量 Token。而用户马上就要把文件发过来了。

**修复：** 建立清晰的约定，区分"等文件"和"立即处理"。作为铁律写入 Schema。

### 9.2 ffmpeg 音频提取最佳参数

`16kHz, 单声道, 64kbps` 是语音转写的最优配置：
- 30MB 视频 → 2MB 音频（16:1 压缩）
- 处理速度：184 倍实时（< 2 秒）
- 语音识别精度无损失
- 成本：30 分钟视频 < $0.01

### 9.3 聊天平台文件发送限制

很多聊天平台不允许在同一条消息中发送文字和文件。这对基于触发词的系统带来了交互挑战。解决方案：实现消息关联机制（在时间窗口内自动关联前后消息）。

### 9.4 上下文压缩后的幻觉记忆

**发现的严重问题：** 在长会话中 LLM 的上下文被压缩后，AI "记得"执行了 3 次入库操作——但实际上根本没有发生。本地文件系统中找不到任何对应记录。

**教训：** 永远以文件系统为准，而不是记忆。灾难恢复审计时，将 `log.md` 条目与 `raw/` 和 `pages/` 中的实际文件逐一核对。

### 9.5 云端文件转存没有捷径

测试发现，从聊天消息文件到云盘之间不存在"一键转存"的 API。即使是界面上的"保存到云盘"按钮，底层也是下载再上传。

**我们的流水线：** 获取消息文件信息 → 下载到本地 → 上传到云盘（超过 20MB 的文件使用分片上传）。

**分片上传的坑：** 上传分片 API 要求每个分片都带 `size` 参数，包括最后一个（可能更小的）分片。漏了这个参数会导致莫名其妙的报错。

### 9.6 实时文档 > 事后文档

在讨论过程中**实时维护设计文档**（而非事后补写）被证明极其有价值：
- 上下文压缩带来的知识丢失为零
- AI 随时可以重读文档恢复上下文
- 文档同时是设计规范和实施记录
- 兼具灾难恢复手册的功能

### 9.7 只追加不修改的文档策略

设计文档采用**只追加（Append-Only）**策略：早期章节可能被后续章节取代。灾难恢复时从附录倒着看，后面的内容覆盖前面的内容。

### 9.8 脚本自安装依赖

每个自动化脚本都应自行安装所需依赖（例如，如果 `trafilatura` 未安装就自动 `pip install`）。这解决了 AI Agent 上下文压缩后"忘记已安装"的老大难问题。

### 9.9 应用创建的资源需要手动赋权

当 AI Agent 的应用程序创建云盘文件夹时，用户默认看不到——需要显式授予权限。创建共享资源后务必立即添加用户权限。

### 9.10 域名兜底必须全面覆盖

调用可能存在多个端点域名的云端 API 时，确保兜底逻辑覆盖**所有**调用点（下载、上传、获取鉴权 Token——不能只覆盖其中一个）。三个独立的 Bug 都追溯到了不完整的域名兜底覆盖。
